## v1.8.1 规划

- P0：电流采样链路评估与整理
  - 目标：先厘清快/慢电流采样的真实差异和必要性，再简化采样相关函数
  - 现状关注点：当前慢采样与快采样在 PAL 层主要区别是 ADC 平均窗口；慢窗口配置意图是覆盖 1 个控制周期，但当前实例 ADC DMA buffer 较小，实际平均窗口可能被底层截断，需先按代码确认真实行为。
  - 拆分：
    1. 梳理 `Sensor_ReadAll`、`Sensor_ReadCurrentOnly`、`Sensor_ReadADC`、`FOC_Platform_ReadPhaseCurrent`、`FOC_Platform_ReadPhaseCurrentFast`、`ADC_ReadPhaseCurrentABOk` 的调用链和执行上下文。
    2. 明确快/慢采样是否都需要保留：若只是平均窗口差异，优先收敛为内部统一读取函数，并用采样 profile 或 avg_count 表达差异。
    3. 处理 `sensor_data` 静态全局在控制任务和 PWM ISR 路径同时更新的风险，避免读写半更新快照。
    4. 修正与代码不一致的注释、配置含义或文档描述。
  - 验收：
    - 电流采样入口语义清晰，控制环路径和电流环 ISR 路径职责明确。
    - 构建 0 error，且不新增 warning。

- P1。0：示波器发送队列重写（先做 osc 波形数据）
  - 目标：控制任务负责按实时数据生成示波器数据帧并写入发送队列；主循环只消费队列并发送，避免主循环直接读电机状态造成频率抖动、读写冲突和波形失真。
  - 设计边界：
    - 示波器发送不再直接读取 `g_motor`、`g_sensor_snapshot` 等运行对象。
    - 控制任务决定采样/入队频率；主循环只根据队列可用数据数量发送 0..N 条消息。
    - 队列建议按单生产者/单消费者设计，生产者为控制任务，消费者为主循环。
  - 队列结构建议包含：
    - 数据数组或帧数组。
    - 读索引、写索引、有效数量。
    - 当前帧发送状态或最后一组数据完成标志。
    - 数据种类/格式标识，用于主循环决定分隔方式和输出格式，可利用宏定义中已存在的掩码相关定义。
    - 溢出计数或丢帧计数，用于后续诊断。
  - 拆分：
    1. 新增示波器数据队列类型和最小 API：初始化、尝试写入、尝试读取、查询数量、清空。
    2. 将 `DebugStream_OutputOscilloscopeFrame` 拆成“数据采集/组帧”和“文本发送”两段。
    3. 在控制任务中按 `osc_report_freq_hz`、`osc_parameter_mask` 生成队列数据。
    4. 在主循环中按队列余量发送有限数量消息，避免一次发送过多阻塞服务任务，要求可宏配置。
    5. 保留现有协议开关和默认行为，不改变上位机帧格式，除非明确同步协议文档。
    6. 修改过程禁止使用兼容式修改法，这是机制彻底重写。但是仍要遵循架构规范
  - 验收：
    - `DebugStream_Process` 或其替代流程不再在发送阶段读取电机运行对象。
    - 示波器启停、频率、参数 mask 仍通过现有协议生效。
    - 主循环被打断时允许丢旧帧或丢新帧，但策略必须明确且可诊断。
    - 构建 0 error，且不新增 warning。

- P1.1：参数回报发送队列重写
  - 目标：将参数/状态回报从命令执行时的直接输出，逐步改为统一的消息队列发送形式，使参数回报与示波器数据流的传递方式一致。
  - 实施前提：优先完成示波器发送队列，并复用或抽象出通用消息队列能力。
  - 现状关注点：当前 `P:X`、单参数读取、状态读取等路径在 L2/C4 执行时直接通过 C5 输出；该任务会牵动 `RuntimeC4Store_ReportAllParams`、`RuntimeC4Store_ReportAllStates`、`RuntimeC5_Output*` 的职责边界。
  - 拆分：
    1. 盘点所有参数/状态/诊断输出路径，区分必须立即反馈的状态字和可排队的文本回报。
    2. 定义参数回报消息结构，避免直接暴露运行时存储内部结构。
    3. 将批量回报 `P:X`、`S:X` 改为逐条入队，主循环分批发送。
    4. 保留命令执行结果状态字的即时反馈能力，避免上位机无法判断命令是否被接收。
    5. 修改过程禁止使用兼容式修改法，这是机制彻底重写。但是仍要遵循架构规范
    6. 读取/回报/队列操作过程属于服务任务，与示波器实时输出监控不同，不要求数据有很高的实时性，数据可以延迟若干个周期，但是仍然不能延时过久，特别是反馈回报
  - 验收：
    - 参数读回不会在 C4 执行阶段进行大量阻塞式文本发送。
    - 批量参数读取不会明显压低控制和服务任务节奏。
    - 协议输出格式保持兼容，或同步更新协议文档。
    - 构建 0 error，且不新增 warning。

  - P1.2：协议操作命令队列化
    - 目标：将输入命令也改为消息队列操作，不再在主循环函数中操作状态机变量，仅使用消息队列，由定时器任务处理相应协议的执行过程
    - 该项未评估

  - 完成P0-P1任务后，应做到：主循环不再参与任何操作数据相关操作，仅生产/消费相应的数据输入输出队列，用于硬件层的输出输出具体操作和协议解析工作，其与控制算法以及运行时完全解耦，

- P2：自定义触发命令与用户回调
  - 目标：新增库协议中的用户自定义触发入口，通过回调函数让实例或用户代码执行自定义操作，扩展协议控制范围。
  - 建议位置：优先接入系统命令 `Y` 通道，新增独立子命令符号；回调由库提供注册 API，实例侧注册具体实现。
  - 设计边界：
    - 回调只在服务任务/主循环命令处理路径执行，不进入 ISR。
    - 回调不得阻塞过久；长操作应只置位请求，由应用任务后续处理。
    - 回调返回明确执行结果，用于协议状态反馈。
  - 拆分：
    1. 定义自定义命令子命令符号和回调类型。
    2. 提供注册、清除和默认空回调行为。
    3. 在 `RuntimeC3_HandleSystemCommand` 中派发自定义命令。
    4. 根据是否允许参数，决定回调签名是否包含 `has_param`、`param_value`、`param_text`。
  - 验收：
    - 未注册回调时返回明确的参数无效或命令无效状态。
    - 注册回调后可通过协议触发一次用户操作并获得状态反馈。
    - 不破坏现有 `Y:R/C/I/G/D/T` 命令。
    - 构建 0 error，且不新增 warning。

- P3：位置传感器采样位置选择
  - 目标：允许用户配置外部位置传感器读取位置：控制环采样用于慢速/阻塞式传感器，电流环或快速路径采样用于快速/非阻塞式传感器。
  - 实施前提：先完成电流采样链路整理，明确快路径中哪些传感器 API 可以安全调用。
  - 设计边界：
    - 当前 `FOC_Platform_ReadMechanicalAngleRad` 未声明 ISR-safe，不能直接默认放入 PWM ISR。
    - 快路径采样只应支持非阻塞传感器，如硬件捕获、霍尔状态缓存、DMA/定时器锁存结果；I2C/SPI 阻塞读取应保留在控制环或服务任务。
  - 拆分：
    1. 新增位置传感器采样位置配置项：控制环采样 / 快速路径采样。
    2. 将角度读取与电流读取解耦，避免 `Sensor_ReadAll` 成为唯一全量入口。
    3. 若支持快速路径，新增或约束平台 API：必须非阻塞、ISR-safe，并能返回最近锁存值。
    4. 在控制算法入口明确使用哪一路角度快照，并处理无效数据降级策略。
  - 验收：
    - 默认配置保持现有行为：控制环读取位置传感器。
    - 慢速位置传感器不会被放入 ISR 路径。
    - 快速采样配置有编译期或运行期保护，避免误用阻塞式平台实现。
    - 构建 0 error，且不新增 warning。
    - 硬件验证项：慢速磁编码器、快速霍尔/缓存式传感器分别验证角度有效性和控制稳定性。

- 结构体及宏裁剪重新整理，按照功能重新划分结构体的分块，明确数据的流向和传入传出点，同时处理注释问题：一句话短中文注释
- 规则冲突问题
- 调整速度环频率后控制效果验证，修复可能的调整频率后时序失配问题（暂缓）
- 新三相硬件实例，旧版两相实例改名，仅作为兼容性留存，开发转移至新硬件，除非需要进行某些必要修复再回到旧版修复（暂缓）
- 三相采样相关测试，完成后进入v2.0.0，准备无感开发(暂缓)
