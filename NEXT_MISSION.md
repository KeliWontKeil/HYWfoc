## 本次目标
v1.0.1 目标：在 v1.0.0 开源基线上，继续进行稳定性与文档治理迭代，保持架构边界清晰与协作流程可维护。

### 优先级与阶段划分

执行顺序说明：先完成库边界与目录结构，再做文档体系治理。依赖前置未完成时，不推进后续文档收敛任务。

定位约束：单电机 FOC 驱动库，集成调试协议与错误处理；不考虑多电机实例扩展。

#### P1（库结构化优先）已完成
P1.1 库边界与仓库结构落地
- 目标：将 L1-L3 + API 作为可移植核心包，其他工程文件作为特定平台示例。
- 交付：
	- 核心库目录与工程示例目录拆分方案（含文件迁移清单）。
	- 将 FOC 库文件独立出项目工程；将现有工程作为 `uvision_EIDE` 示例。
	- 示例工程目录预留为“硬件/软件”双目录：硬件目录后续补充，软件目录承载当前工程。
	- API 分为两类：
		- 核心侧提供空实现/接口约束文件（放在示例外）。
		- 各示例提供本平台 API 实现文件（放在示例内）。
	- L1-L3 作为公共依赖库，示例工程仅引用，不做拷贝副本。
- 确认方式：输出目录草案 + 迁移清单 + 依赖关系图（本阶段不设硬性验收条款）。

#### P2（开源基础合规）已完成（v1.0.0 发布门禁通过）
P2.1 许可证与第三方依赖合规
- 目标：满足开源发布的法律与分发要求，使用 MIT 协议。
- 交付：
	- 项目许可证文件与版权声明。
	- 第三方组件清单与许可证说明（含 `Firmware` 目录依赖）。
- 确认方式：仓库内可追溯的许可证与第三方声明文件齐备。

#### P3（文档体系治理，依赖 P1 完成后进行）已完成（v1.0.0 发布门禁通过）
P3.1 分层边界静态审计与文档同步
- 目标：基于最终库结构复核 L1/L2/L3/L4 + Special Layer 依赖方向，消除边界歧义。
- 交付：
	- 依赖边界审计清单（违规项、风险等级、建议动作）。
	- 同步更新 `docs/architecture.md` 与 `docs/structure-and-dependency-tree.md`。
- 确认方式：审计清单与结构图一致即可，不单设流程型验收。

P3.2 SSOT 收敛与交叉引用规则
- 目标：协议、默认参数、调度频率、通道映射在各文档保持一致。
- 交付：
	- 文档一致性检查清单（版本号、默认参数、通信通道、调度频率、故障策略）。
	- `README.md`、`docs/*`、`CHANGELOG.md` 的交叉引用规则与执行说明。
- 确认方式：完成一致性清单并消除已知冲突项。

P3.3 工程目录职责与提交边界文档化
- 目标：明确“源码/构建产物/IDE工程文件/文档”边界，减少无关噪声变更。
- 交付：
	- 目录职责说明与提交流程边界约定。
	- 明确产物文件不进入版本发布提交的规则。
- 确认方式：规则文件与 `.gitignore`/发布流程描述一致。

P3.4 规则与流程文档标准化
- 目标：开发规则、发布流程、执行步骤保持一致，避免流程漂移。
- 交付：
	- 每次流程变化后同步更新中英文规则与开发文档。
	- 增加“待人工验证”标记机制，用于当前无法自动验收条目。
- 说明：移除“新人可仅凭文档完成完整构建与发布演练”作为本阶段硬验收项。

P3.5 项目重命名与注释清晰化
- 目标：完成对外命名口径统一与文档角色分层，支撑开源展示与用户上手。
- 命名基线：
	- 英文名：HYWfoc（How Y-axis Works FOC）
	- 中文名：何易位FOC（允许“何易位foc”作为文案变体）
	- 释义：电机转子如何改变自己的位置（隐喻磁场定向控制）
- 交付：
	- 根 `README.md`：对外宣传/项目介绍（中文主叙事，保留少量英文副标题）。
	- 实例 `examples/GD32F303_FOCExplore/README.md`：平台与使用方式用户文档。
	- `docs/README.md`：库级文档索引与边界规则与根 README 保持一致。
	- `CHANGELOG.md`：记录命名与文档角色重构（仅文档变更，不混入代码变更描述）。
- 依赖关系：
	- 依赖 P3.2（SSOT 收敛）确保名称、版本、任务状态在多文档一致。
	- 依赖 P3.4（规则标准化）确保流程描述在中英文规则与开发文档一致。
- 确认方式：
	- 口径一致：上述 4 处文档中的命名、释义、版本基线一致。
	- 角色清晰：根 README 不下沉实例细节；实例 README 不复写库级规则。
	- 引用有效：README/docs/NEXT_MISSION/CHANGELOG 相互引用可达，无断链。
